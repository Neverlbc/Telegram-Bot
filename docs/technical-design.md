# 技术方案文档

## 1. 系统架构

```
┌──────────────┐
│  Telegram    │
│  API Server  │
└──────┬───────┘
       │ HTTPS (Webhook / Polling)
       ▼
┌──────────────────────────────────────────────────┐
│                Bot 应用层                         │
│                                                  │
│  ┌─────────┐  ┌──────────┐  ┌────────────┐      │
│  │ Handler  │  │ 中间件链  │  │ FSM 引擎   │      │
│  │ (路由)   │  │ i18n/DB/ │  │ (状态管理)  │      │
│  │         │  │ throttle │  │            │      │
│  └────┬────┘  └──────────┘  └────────────┘      │
│       │                                          │
│  ┌────▼──────────────────────────────────────┐   │
│  │            Service 业务层                  │   │
│  │  聚水潭对接 │ 物流查询 │ AI回复 │ 通知      │   │
│  └────┬──────────┬──────────┬────────────────┘   │
└───────┼──────────┼──────────┼────────────────────┘
        │          │          │
   ┌────▼────┐ ┌───▼────┐ ┌──▼───────────────┐
   │  MySQL  │ │ Redis  │ │   外部 API        │
   │  8.0    │ │  7.x   │ │ 聚水潭/CDEK/      │
   └─────────┘ └────────┘ │ RU Post/Cainiao  │
                           └──────────────────┘
```

### 组件职责

| 组件                 | 职责                                          |
| -------------------- | --------------------------------------------- |
| **Bot 应用层** | 接收 Telegram 消息，路由到对应 Handler        |
| **中间件链**   | 频率限制 → 数据库会话注入 → 多语言加载      |
| **FSM 引擎**   | 管理多步骤会话状态（物流查询、批发下单）      |
| **Service 层** | 封装业务逻辑，Handler 不直接操作外部 API      |
| **MySQL**      | 持久化存储（用户、商品、订单、工单）          |
| **Redis**      | FSM 状态存储、AI 回复触发计数、频率限制、缓存 |
| **聚水潭 API** | 查询订单发货状态（已发货/未发货）             |
| **物流 API**   | 查询物流轨迹（CDEK、RU Post、Cainiao）        |

## 2. 技术选型

### 2.1 Bot 框架：aiogram 3.x

**选型理由：**

- **原生异步**：基于 asyncio，天然适配多个外部 API 并发调用（聚水潭 + 物流查询）
- **内置 FSM**：StatesGroup + State 开箱即用，适配物流查询、批发下单等多步骤流程
- **Router 分模块**：每个功能模块独立 Router，售后/物流/客服互不干扰
- **中间件体系**：支持请求级中间件，适合做 i18n 注入、DB 会话管理、频率限制

### 2.2 数据库：MySQL 8.0

**选型理由：**

- 华为云 RDS for MySQL 生态成熟，运维文档和中文资料丰富
- 本项目数据操作以简单 CRUD 为主，MySQL 性能优秀
- 低规格云服务器上内存占用更友好

### 2.3 缓存/状态：Redis 7.x

| 用途         | 说明                                                     |
| ------------ | -------------------------------------------------------- |
| FSM 状态存储 | aiogram `RedisStorage`，会话状态持久化，Bot 重启不丢失 |
| AI 回复计数  | 记录同一用户对同一订单的查询次数，触发 5 次转人工        |
| 频率限制     | 基于 `INCR + EXPIRE` 的滑动窗口计数器                  |
| 商品缓存     | 热门商品列表缓存，减少数据库查询                         |
| 物流缓存     | 物流查询结果短期缓存（避免频繁调用外部 API）             |

### 2.4 ORM：SQLAlchemy 2.0 (async)

- 使用 `AsyncSession` + `aiomysql` 驱动
- 2.0 风格声明式模型，类型提示友好
- 配合 Alembic 管理数据库迁移

### 2.5 代码质量工具

| 工具           | 用途          |
| -------------- | ------------- |
| ruff           | lint + 格式化 |
| mypy           | 静态类型检查  |
| pytest         | 测试框架      |
| pytest-asyncio | 异步测试支持  |

## 3. 核心模块设计

### 3.1 售前咨询 — 自动回复设计

```
用户点击「售前咨询」
       │
  ┌────┼────────────┐
  ▼    ▼            ▼
查看  配送说明    常见问题
商品清单   │            │
  │    展示静态    展示 FAQ
  ▼    配送说明    列表（分页）
选择分类
  │
  ▼
选择产品
  │
  ▼
选择规格（工业/开源/特殊）
  │
  ▼
查找预设自动回复
  │
  ├── 有预设 → 自动回复（价格/规格/说明）
  │               + 「联系人工」按钮
  └── 无预设 → 转特定人工客服
```

**实现要点：**

- 商品清单复用 `categories` + `products` 表，规格选项通过 `product_variants` 或 `products.auto_reply_*` 字段存储
- 自动回复内容支持三语（中/英/俄），管理员可通过后台编辑
- 「无预设」的判断：该产品+规格组合的 `auto_reply_zh` 字段为空
- 自动回复展示后，始终保留「联系人工」入口，用户可随时升级到人工
- 配送说明和常见问题为静态内容，存储在 `faq_items` 表，支持分类（`type` 字段区分 delivery/faq）

### 3.2 多语言中间件

```
用户消息 → I18nMiddleware → 从 DB 读取用户语言偏好 → 注入 _() 翻译函数 → Handler
```

**实现要点：**

- 中间件在每次请求时从数据库加载用户语言设置
- 首次用户（未设置语言）根据 Telegram `language_code` 自动推断
- 翻译函数 `_()` 注入到 handler 参数中
- 翻译文件格式：GNU gettext `.po/.mo`
- 目录结构：`locales/{zh,en,ru}/LC_MESSAGES/messages.po`

### 3.2 售后支持 — 聚水潭 ERP 对接

```
用户点击「售后支持」
       │
       ▼
  查询订单状态（是否发货）
       │
       ├── 需要授权聚水潭 ──→ 引导用户授权
       │
       ▼
  调用聚水潭 API 查询
       │
  ┌────┴────┐
  ▼         ▼
已发货    未发货
  │         │
  ▼         ▼
授权物流   AI 自动
跟踪网站   拖延回复
自动回复       │
客户状态       ▼
          同一用户触发 5 次
               │
               ▼
          转特定人工
```

**实现要点：**

- 聚水潭 API 对接封装在 `bot/services/jushuitan.py`
- 使用 OAuth2 授权模式，token 缓存在 Redis
- 查询结果缓存 5 分钟，避免频繁调用
- 已发货 → 自动跳转到物流查询流程
- 未发货 → AI 生成安抚/拖延回复
- AI 回复触发计数存 Redis：`ai_reply:{user_id}:{order_id}`
- 触发 5 次后自动转接特定人工客服

### 3.3 物流查询 — 多物流商适配

采用**策略模式**，将物流查询逻辑与 Bot 逻辑解耦：

```
LogisticsTracker (ABC)
├── query_by_tracking_number(tracking_no) → TrackingResult
└── get_tracking_url(tracking_no) → str

具体实现：
├── CDEKTracker          # CDEK 快递
├── RUPostTracker        # 俄罗斯邮政
├── CainiaoTracker       # 菜鸟物流
└── AirFreightTracker    # 空运

LogisticsTrackerFactory
└── get_tracker(carrier: str) → LogisticsTracker
```

**物流查询流程：**

```
用户点击「物流查询」
       │
       ▼
  选择发货地
       │
  ┌────┴────┐
  ▼         ▼
莫斯科    中国发货
发货         │
  │         ▼
  ▼      输入跟踪号
选择物流商      │
(CDEK/RU Post/  ▼
 Cainiao/空运) 授权物流跟踪网站
  │         自动回复客户状态
  ▼
输入跟踪号
  │
  ▼
授权物流跟踪网站
自动回复客户状态
```

**实现要点：**

- 新增物流商只需实现接口并注册到工厂
- 查询结果缓存在 Redis（TTL 30 分钟），避免重复调用
- 支持自动识别跟踪号格式推断物流商
- 查询失败时给出友好提示，建议用户检查跟踪号或转人工

### 3.4 AI 自动回复

```
用户查询未发货订单
       │
       ▼
  Redis 读取计数
  ai_reply:{user_id}:{order_id}
       │
  ┌────┴────┐
  ▼         ▼
 < 5 次    ≥ 5 次
  │         │
  ▼         ▼
AI 生成    转特定人工
拖延回复   （附带历史记录）
  │
  ▼
Redis INCR 计数
```

**实现要点：**

- AI 回复内容根据已等待天数动态调整语气
- 回复需多语言支持（根据用户语言偏好）
- 计数 key 设置合理 TTL（如 7 天），过期自动重置
- 转人工时附带用户查询历史和订单信息，方便客服处理

### 3.5 设备支持 — 模块设计

```
用户点击「设备支持」
       │
  ┌────┼──────────────┐
  ▼    ▼              ▼
配置  设备问题      更多选项
序列号    │              │
  │   ┌──┼──┬───┐  ┌───┼───┐
  ▼   ▼  ▼  ▼   ▼  ▼   ▼   ▼
绑定  固件 硬件 软件 远程 固件 硬件 远程
数据库 类  类  类  类  类  类  类
  │    └──┴──┴───┘  └───┴───┘
  ▼         │               │
字段查询   提供数量         提供授权码
          型号+序列号      +说明
               │               │
               ▼               ▼
          授权发厂家       发相关人员
               │               │
         ┌─────┴────┐    ┌─────┴────┐
         ▼          ▼    ▼          ▼
      机器人      转人工  机器人    转人工
      完成处理           完成处理
```

**实现要点：**

- **查询序列号**：需绑定一个外部数据库，通过字段查询序列号信息
- **设备问题**（4 类型：固件/硬件/软件/远程）：
  - 收集用户提供的数量、型号、序列号
  - 获取厂家授权后发送对应产品信息
  - 机器人完成处理 或 转人工处理
- **更多选项**（3 类型：固件/硬件/远程）：
  - 收集用户提供的授权码和问题说明
  - 发送给对应处理人员
  - 机器人完成处理 或 转人工处理
- 设备类型通过 `DeviceIssueType` 枚举定义，便于扩展

### 3.6 菜单导航与回退

主菜单共 **5 个入口**，售后支持包含 3 个子模块：

```
主菜单（5项）
├── 售前咨询 → 查看商品清单 / 配送说明 / 常见问题
├── 售中下单 → 批发订单 → 留言 → 转人工
├── 售后支持（3个子模块）
│   ├── 订单状态查询 → 聚水潭 ERP → 已发货/未发货
│   ├── 物流查询 → 选择发货地 → 选物流商 → 输入跟踪号
│   └── 设备支持 → 查询序列号 / 设备问题 / 更多选项
├── 客服合作 → 商务合作 / 批发咨询 / 转人工
└── 设置 → 语言切换
```

**实现要点：**

- 每个菜单页面通过编辑消息（`edit_message_text`）切换，避免刷屏
- 除语言选择和主菜单外，**每个页面底部必须同时提供**「返回上级菜单」和「返回主菜单」两个按钮
- FSM 输入等待状态同样提供两个返回按钮（用户点击即退出 FSM 并跳转）
- 使用 `CallbackData` 携带必要上下文（当前分类 ID、页码、来源页等）
- `NavCallback(action="back")` 返回上级，`NavCallback(action="home")` 返回主菜单

### 3.6 人工客服转接

```
用户请求转人工 → 创建客服工单 → 通知客服群组 → 客服认领
       │                                        │
       ▼                                        ▼
  Bot 告知用户                             建立 1v1 消息转发
  "正在转接..."                            （Bot 中转消息）
                                                │
                                                ▼
                                          客服关闭工单
                                          Bot 通知用户
```

**触发转人工的场景：**

- 用户主动点击「转接人工」
- AI 自动回复触发 5 次后自动转接
- 批发订单留言后自动转接
- 售后问题无法自动解决时

**实现要点：**

- 客服群组为指定的 Telegram Group，客服人员在群内认领工单
- Bot 作为中转，将用户消息转发给客服，客服回复转发给用户
- 工单状态：待接入 → 处理中 → 已关闭
- 超时无人认领自动提醒管理员

## 5. 安全设计

### 5.1 密钥管理

- Bot Token、数据库密码、聚水潭密钥、物流 API 密钥统一通过 `.env` 文件管理
- `.env` 加入 `.gitignore`，不入库
- 生产环境通过 Docker secrets 或华为云密钥管理注入

### 5.2 外部 API 安全

- 聚水潭 API 调用需 OAuth2 授权，token 定期刷新
- 物流 API 密钥定期轮换
- 所有外部 API 调用设置超时（默认 10 秒）
- 外部 API 异常时降级为友好提示，不暴露错误详情

### 5.3 频率限制

- 全局限制：同一用户每秒最多 2 次请求
- 物流查询限制：同一跟踪号每 30 分钟查询一次（缓存）
- 基于 Redis INCR + EXPIRE 实现

### 5.4 管理员权限

- 管理员 ID 通过环境变量 `ADMIN_IDS` 配置
- 管理后台命令（商品管理、用户管理、统计）仅管理员可用
- 通过中间件在 handler 前校验权限
